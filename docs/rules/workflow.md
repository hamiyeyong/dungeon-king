# 작업 워크플로우

## 기본 작업 흐름 (모든 기능/버그/리팩토링)

**별도 지시 없이도** 아래 흐름을 자동 실행한다:

1. `EnterWorktree` — 브랜치 + 워크트리 생성 후 작업
2. 구현 완료 → `/create-pr` 스킬로 PR 생성
3. PR 생성 직후 → 머지 & 워크트리 정리 (아래 섹션 참조)

**Why:** 1인 개발이지만 모든 스펙 기반 변경은 브랜치 + PR 흐름이 원칙 (전역 CLAUDE.md). 매번 "워크트리로 할까요?" 확인이 불필요한 마찰.

**예외:** 오타·주석·포맷 등 trivial 변경은 main 직접 커밋 허용 (워크트리 불필요).

## PR 머지 & 워크트리 정리

PR 생성 후 **자동으로 아래 순서를 실행**한다 (매번 확인 불필요):

1. `gh pr merge <번호> --squash` — PR 스쿼시 머지
2. `git push origin --delete <브랜치>` — 원격 브랜치 삭제
3. `ExitWorktree(action: "remove", discard_changes: true)` — 워크트리 제거

**Why:** 1인 개발 프로젝트라 PR 생성 = 즉시 셀프 머지가 기본 흐름. 매번 "머지할까요?" 확인이 불필요한 마찰.

**예외:** 사용자가 "draft로 올려줘" / "리뷰 후 머지" 등 명시적으로 보류 요청 시 머지 생략.

## 토큰 절약 원칙

- 파일 읽기 전 줄번호 범위를 특정할 수 있으면 전체 읽기 금지
- 서브에이전트는 독립 병렬 작업 2개 이상일 때만 호출
- 응답에 요약/재설명 붙이지 말 것 (diff가 답임)
- 탐색성 질문("어떻게 하면 좋을까?")에 긴 분석 응답 금지 — 2~3줄 추천 + 사용자 확인 후 진행
